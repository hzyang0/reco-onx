# V20：容量压测、百分位延迟与 SLO 门禁

V19 之前回答的是“功能能不能工作”；V20 回答“持续来流量时，系统在什么边界内仍然正确”。压测的目标不是制造一个最大的 QPS 数字，而是找到满足延迟、成功率和推荐质量的安全容量。

## 1. 为什么只看平均耗时会骗人

10 个请求中 9 个 100ms、1 个 2s，平均值 290ms 看起来尚可，但每十个用户就有一个等两秒。百分位从用户分布看问题：

- P50：一半请求不超过它，代表典型体验；
- P95：95% 请求不超过它，常用于服务目标；
- P99：观察长尾，容易暴露 GC、排队和下游抖动；
- Max：对小样本很敏感，只作诊断，不宜单独作为稳定 SLO。

V20 使用 nearest-rank：排序后取 `ceil(P/100 × N)` 对应样本，并用 JUnit 覆盖边界。

## 2. SLI、SLO、SLA 分别是什么

- **SLI** 是实际测量值，例如 P95=270ms、成功率=100%。
- **SLO** 是内部目标，本项目定义成功率≥99%、P95≤1200ms、召回兜底率≤5%。
- **SLA** 是对客户的合同承诺，违约可能赔偿；本学习项目没有 SLA。

这里特意加入兜底率：接口即使 HTTP 200，也可能只拿到部分召回再用热门结果补足 10 条。若只看 200 成功率，会把推荐质量损失藏起来。

## 3. 压测器如何工作

`CapacityBenchmarkApplication` 是项目自带的闭环负载生成器：每个虚拟用户收到响应后立即发下一个请求。它会先 warm-up，再在固定时间内记录每个样本的耗时、正确性和 FALLBACK，按并发 1/2/4/8 做阶梯搜索。

闭环模型适合本地容量探索，但生产压测还要增加固定到达率的开环模型，否则服务变慢时客户端也变慢，可能低估排队压力。

压测用户在 100 个稳定 ID 中循环，既覆盖 Redis miss→hit，也避免每次都制造完全冷的数据。

## 4. 三层超时为什么必须递增

首轮真实压测发现，并发 1 都偶发兜底。根因不是 HTTP 错误，而是正常 gRPC 耗时贴近三层旧预算：

```text
gRPC deadline 70ms < 韧性包装 80ms < fan-out 总截止 120ms
```

本地容器调度稍有抖动就会误杀。V20 将三层做成环境变量，并在容量验收中使用：

```text
GRPC_DEADLINE_MS=100
RECALL_TIMEOUT_MS=120
RECALL_FANOUT_TIMEOUT_MS=150
```

原则是内层先超时并返回，外层留出清理、聚合和传播时间。不能把所有层设成相同数字，也不能无限放大掩盖真正的慢请求。

## 5. 本机实测结果

2026-07-16 在本机 Docker Desktop 七容器环境、每档 warm-up 2 秒并测量 8 秒，得到：

| 并发 | QPS | 成功率 | 兜底率 | P50 | P95 | P99 | SLO |
|---:|---:|---:|---:|---:|---:|---:|:---:|
| 1 | 4.57 | 100% | 0% | 218ms | 301ms | 302ms | PASS |
| 2 | 9.33 | 100% | 1.32% | 217ms | 270ms | 302ms | PASS |
| 4 | 13.19 | 100% | 50.46% | 287ms | 458ms | 503ms | FAIL |
| 8 | 12.71 | 100% | 47.12% | 608ms | 799ms | 980ms | FAIL |

因此当前**最大通过并发为 2，安全吞吐约 9.33 QPS**。并发 4 的表面吞吐更高，但一半请求发生下游兜底，不能算有效容量；并发 8 的吞吐反而下降、延迟继续上升，已进入饱和区。

这是本地短时基准，不是生产承诺。硬件、容器限额、JVM、网络、样本时长和数据分布改变后必须重新测量，并留容量余量。

## 6. 一键最终验收

```powershell
.\scripts\run-capacity-test.ps1
```

脚本会：

1. 执行 54 项测试并构建 V20；
2. 启动 gateway、三路召回、collector、配置中心、Redis 共七个容器；
3. 预热后执行 1/2/4/8 容量阶梯；
4. 输出 `target/capacity-v20/summary.json`、`results.csv`、`report.md`；
5. 再用 P95≤1ms 的不可能门槛，证明 SLO gate 会真实失败；
6. 停止 live，验证 HTTP 仍成功但质量 FALLBACK 可被识别；
7. 恢复 live，确认三路召回重新达到 25 条；
8. 输出七容器 CPU/内存快照并自动清理。

手工运行压测器：

```powershell
java -cp target/mini-reco-access-layer-0.1.0-SNAPSHOT.jar `
  com.interview.minireco.capacity.CapacityBenchmarkApplication `
  http://localhost:18093 2 8 '1,2,4,8' target/my-capacity-report
```

可用环境变量覆盖 `SLO_MIN_SUCCESS_RATE`、`SLO_MAX_FALLBACK_RATE`、`SLO_MAX_P95_MS` 和 `LOAD_REQUEST_TIMEOUT_MS`。

## 7. 从结果怎样继续优化

并发 4 的主要失败项是 fallback，而不是 HTTP error 或 P95。下一步应先按指标拆解 goods/live/ad 哪一路超时，再考虑：

- 调整每路 bulkhead 线程池与队列，而不是盲目扩大所有线程；
- 检查 gRPC 服务是否在阻塞执行线程上做耗时操作；
- 给不同下游设置基于历史分位数的 deadline；
- 扩容召回实例或按来源隔离资源；
- 延长测量到 30 分钟以上，加入 GC、CPU、连接池、Redis 和网络指标；
- 用二分搜索细化并发 2～4 之间的边界，并按结果预留 30%～50% 容量余量。

## 8. 面试表达

> 我没有把 HTTP 200 当成成功的全部定义，而是把返回条数和多路召回 FALLBACK 纳入质量 SLI。自研闭环压测器完成 warm-up、并发阶梯、P50/P95/P99、QPS 和 JSON/CSV/Markdown 报告，并让 SLO 失败以非零码阻断流水线。首轮发现并发 1 也偶发兜底，定位为 70/80/120ms 三层 deadline 过紧且不可配置，改成可配置的 100/120/150ms 递增预算。最终并发 2 在 9.33 QPS、P95 270ms、兜底 1.32% 下通过；并发 4 虽 QPS 更高但兜底约 50%，因此安全容量仍判为 2。
