# V21：持久化配置中心——从“内存变量”到“重启不丢”的发布系统

## 1. 这一版解决什么问题

V18 的动态配置中心已经能发布灰度比例、影子流量比例和降级级别，但配置只存在 JVM 内存里。进程一重启，版本就回到初始值 1。线上服务会因为发布记录消失而使用错误策略，审计记录也无法追查。

V21 的目标可以用一句话概括：**已经返回发布成功的配置，即使配置中心随后立刻重启，也必须找得回来。**

验收标准不是“有一个文件”，而是四件事同时成立：

1. 发布成功前，配置已经持久化；
2. 配置中心重启后，恢复最新版本；
3. 最近 20 条审计历史仍然存在；
4. 恢复之后还能继续发布下一个连续版本。

## 2. 先理解五个前置概念

### 2.1 内存和磁盘

Java 对象存在内存里，读写快，但进程退出后消失。文件存在磁盘或 Docker Volume 中，速度较慢，但进程重启后仍在。动态配置的查询很频繁，所以正常读仍从内存完成；只有发布更新时写磁盘，启动时再从磁盘恢复。

### 2.2 快照

一次完整配置叫一个 `RuntimeConfigSnapshot`，包含：

- `version`：连续递增的版本号；
- `newPipelinePercent`：新链路灰度比例；
- `shadowPercent`：影子流量比例；
- `degradationLevel`：降级级别；
- `updatedBy` 和 `updatedAt`：谁在什么时候发布。

快照是不可变对象。发布 V3 时不会修改 V2，而是创建新的 V3，因此恢复和审计都更容易。

### 2.3 Journal（追加日志）

项目没有每次覆盖整个配置文件，而是每发布一个版本就在文件末尾追加一行 JSON：

```text
{"version":1,...}
{"version":2,...}
{"version":3,...}
```

这叫 append-only journal。启动时从第一行依次回放，就能得到历史和最新状态。它类似数据库 WAL、消息日志和 Git 提交历史的简化版。

### 2.4 fsync / force

操作系统接收到写文件请求后，数据可能暂时只在缓存中。`FileChannel.force(true)` 要求把数据尽量同步到持久化设备。代码只有在 `append` 和 `force` 成功后，才把新快照发布到内存并返回成功。

这条顺序非常重要：

```text
校验 expectedVersion -> 构造新快照 -> 追加并刷盘 -> 更新内存 -> 返回成功
```

如果先更新内存再写文件，写盘失败时，调用方可能看到一个重启后消失的“假成功版本”。

### 2.5 Docker Volume

容器的可写层随容器删除而消失。`compose.yaml` 把命名卷 `config-data` 挂载到 `/data`，并把日志写到 `/data/config-journal.jsonl`。因此重启或重建配置中心容器时，数据仍由 Volume 保存。

## 3. 代码地图

### `ConfigJournal.java`

持久化抽象，只约定三件事：加载全部快照、追加快照、描述存储位置。业务层依赖接口，不依赖具体文件实现，因此单测可替换为内存或故障实现。

### `InMemoryConfigJournal.java`

本地默认和普通单元测试使用。它不解决重启持久化，但避免每个测试都创建文件，也保留了不开启文件存储时的兼容行为。

### `FileConfigJournal.java`

真正的 JSON Lines 文件实现：

- `load()` 顺序解析每一行，并验证版本必须为 1、2、3……；
- `append()` 用追加方式写一整行，然后 `force(true)`；
- 中间行损坏或版本断档时拒绝启动，避免静默使用错误配置；
- 仅最后一行损坏时，认为可能是宕机导致的半次写入，保留之前的完整版本并重写日志。

### `DynamicConfigStore.java`

它是领域层的配置仓库：

- 构造时调用 journal 回放；
- 空日志时创建并持久化 bootstrap V1；
- 更新时检查乐观锁；
- 先 `journal.append(next)`，成功后才更新 `AtomicReference`；
- 内存中只保留最近 20 条历史供接口查询。

### `ConfigCenterApplication.java`

它是 HTTP 接入层。环境变量 `CONFIG_STORE_PATH` 存在时创建文件 Journal，否则使用内存 Journal。`/health` 同时返回当前版本和存储位置；落盘失败返回 HTTP 503，告诉发布方这是服务暂不可用，而不是参数写错。

## 4. 一次发布请求如何执行

假设当前版本是 3，发布方请求更新到 4：

```text
POST /api/config?expectedVersion=3&...
        |
        v
解析并校验参数
        |
        v
当前版本是否仍为 3？ -- 否 --> 409 Conflict
        |
       是
        v
构造不可变 V4 快照
        |
        v
追加 JSON + force(true) -- 失败 --> 503，内存仍是 V3
        |
       成功
        v
内存切换为 V4，加入审计历史
        |
        v
200 OK；网关下一轮轮询应用 V4
```

这就是乐观锁和 durable-before-publish（持久化先于发布）的组合。

## 5. 为什么只自动修复最后一行

进程可能在一行 JSON 只写了一半时断电，这种损坏只能发生在正在追加的最后一行，所以可以安全丢弃这次未完成发布。

如果第 5 行损坏，但第 6 行还存在，它就不再像一次普通的末尾宕机，更可能是磁盘、人工编辑或数据传输问题。此时自动跳过会掩盖数据事故，所以系统选择启动失败，要求人工检查。

版本断档同理。日志出现 V1、V2、V4 时，系统不能猜测 V3 内容，必须拒绝恢复。

## 6. 单元测试分别证明什么

`FileConfigJournalTest` 覆盖三个关键场景：

1. 创建第二个 Store 模拟进程重启，验证版本和历史恢复；
2. 人工制造截断末行，验证修复后还能追加且再次重启仍可读取；
3. 人工制造版本断档，验证系统拒绝带病启动。

`DynamicConfigStoreTest` 还使用一个故意抛异常的 Journal，验证落盘失败时内存版本不变化。这就是依赖倒置带来的可测试性：不需要真的弄坏一块硬盘，也能稳定复现故障。

## 7. 本地动手验收

先确保 Docker Desktop 已启动，然后执行：

```powershell
.\scripts\run-persistent-config-demo.ps1
```

脚本会自动完成：58 项测试、镜像构建、六服务启动、发布 V2/V3、真实重启配置中心、恢复 V3 与三条历史、继续发布 V4，最后清理容器与 Volume。

想保留环境观察：

```powershell
.\scripts\run-persistent-config-demo.ps1 -KeepRunning
docker compose exec config-center sh -c "wc -l /data/config-journal.jsonl"
docker compose down --volumes
```

注意：最后的 `--volumes` 会删除学习环境的数据卷，生产环境不能随意执行。

## 8. 面试时怎么讲

可以按“问题—方案—一致性—容灾—验证”回答：

> 原配置中心是内存存储，进程重启会丢失灰度策略和审计记录。我抽象了 ConfigJournal，并实现 JSON Lines 追加日志；更新时先校验乐观锁，再落盘并 force，最后发布到内存，避免返回成功但重启丢失。启动时顺序回放并校验连续版本，只容忍宕机可能造成的末行截断，中间损坏直接拒绝启动。Docker 用命名卷保存日志。我通过单测覆盖落盘失败、版本断档和末行修复，又真实重启容器验证 V3 和历史恢复，并在重启后继续发布 V4。

如果面试官追问“为什么不用 MySQL”，可以回答：本项目先用最小实现把一致性边界讲清楚；单机追加日志适合学习和低写入量，但生产多副本会需要数据库、Raft/etcd 或成熟配置中心来解决主从、选主、并发写和备份恢复。知道方案边界，比声称一个文件能解决所有生产问题更重要。

## 9. 这一版仍未解决的问题

- 多个配置中心副本同时写同一日志；
- 发布接口没有身份认证和权限控制；
- 密钥轮换、重放攻击防护和发布审批；
- 自动化发布门禁与最终故障演练。

这些正是 V22 要补齐的内容。
