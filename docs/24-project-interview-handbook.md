## 0. 面试时先建立一条主线

这个项目模拟电商推荐接入层。它承接商城等上游流量，准备请求参数和用户特征，并行调用商品、直播和广告召回，补充实时特征并过滤无效 Item，再调用混排，最后统计、记录日志并返回结果。

完整主链路：

```text
HTTP 请求
  -> 参数校验、AB 参数、地址、用户特征
  -> 商品/直播/广告三路并行召回
  -> 在线价格/库存/状态特征
  -> 过滤
  -> 混排
  -> 后处理、指标、日志、Trace
  -> HTTP/Protobuf 响应
```

工程治理主线：

```text
Big Map 串行代码
  -> 类型安全 Context
  -> 统一 Item 和属性 Map
  -> 小算子
  -> DAG 调度
  -> 并行 DAG
  -> 超时、重试、熔断、隔离、兜底
  -> 灰度与影子双跑
  -> Protobuf + gRPC 多进程
  -> 日志、指标、Trace、告警
  -> Docker + Kubernetes
  -> 动态配置 + Redis 缓存
  -> 容量压测 + SLO
  -> 配置持久化 + 管理面安全 + CI
```

面试中每个技术点都用四句话组织：

1. 原来有什么问题；
2. 为什么选择这个方案；
3. 方案如何工作；
4. 如何验证、有什么边界。

## 1. 推荐系统与接入层基础

### 1.1 推荐系统为什么分召回、粗排、精排和混排？

标准回答：候选全集可能有千万甚至上亿条，无法对每条内容运行昂贵模型。召回先用多种低成本策略快速缩小候选集；粗排继续压缩；精排用更复杂模型计算相关性；混排综合商品、直播、广告以及业务规则，生成最终列表。这是一种分阶段漏斗，在效果、延迟和计算成本之间平衡。

项目落点：项目重点实现接入层、三路召回、过滤和混排，没有实现真实机器学习模型。

### 1.2 什么是多路召回？

标准回答：不同召回通道负责发现不同类型或不同兴趣信号的候选，例如商品协同过滤、直播兴趣和广告定向。多路召回提高覆盖率和多样性，但会带来协议不一致、重复 Item、延迟放大和部分失败处理问题。

项目落点：`GoodsRecallService`、`LiveRecallService`、`AdRecallService`，以及 `ParallelRecallFanout`。

### 1.3 为什么需要接入层？

标准回答：接入层隔离上游场景和下游算法服务，统一完成参数治理、特征准备、召回编排、协议转换、稳定性保护、灰度、监控和结果后处理。算法服务专注策略，业务入口不必了解每一路服务细节。

### 1.4 接入层和 API Gateway 有什么区别？

API Gateway 更偏通用流量治理，例如鉴权、路由、限流和 TLS；推荐接入层包含推荐领域编排，例如 AB 参数、用户特征、多路召回、过滤和混排。两者可以同时存在，前者是通用基础设施，后者是业务服务。

### 1.5 什么是在线特征？为什么召回后还要再取？

召回携带的价格、库存、商品状态可能已经过期。在线特征服务提供更实时的数据，用于混排或过滤。其耗时通常随 Item 数量增加，因此扩大召回量可能显著拉高延迟。

项目落点：`OnlineFeatureService`、`OnlineFeatureOperator`。

### 1.6 为什么“在线特征和混排并行，之后再过滤”能节省约 120ms？

原链路是在线特征约 120ms，加混排约 100ms，串行关键路径接近 220ms。两者没有严格数据依赖时可以并行，关键路径变为 `max(120, 100)`，理论节省约 100ms 以上。代价是过滤挪到混排后，少量无效 Item 会破坏混排原顺序，需要用候补结果填充，并用数据确认被过滤比例足够小。

高频追问：为什么不能只说“并行一定更快”？因为并行有线程调度、上下文切换、资源争用和合并开销，而且只有无依赖任务才能并行。

## 2. 重构、复杂度与设计原则

### 2.1 什么是软件熵？

软件熵不是一个严格运行指标，而是描述系统随着需求叠加出现耦合、重复、隐式依赖和认知成本上升。典型现象是一个小需求需要改很多文件、害怕影响未知链路、定位问题困难、测试无法隔离。

### 2.2 为什么重构背景不是性能问题也合理？

工程效率、可测试性和变更风险同样是系统成本。如果“加一个特征打点”都需要大量人力，说明架构阻碍业务迭代。重构首先降低认知和变更成本，性能改善可能只是附加收益。

### 2.3 如何避免“大重构”失控？

- 保持外部行为和接口尽量不变；
- 分代码结构、执行图、运行治理逐步迁移；
- 新旧链路影子双跑并做结果 Diff；
- 小比例灰度，指标异常能一键回滚；
- 每一层补单测、指标和故障演练；
- 明确完成标准，不把顺手优化无限扩大。

### 2.4 单一职责原则是什么？

一个类或算子应该只有一种主要变化原因。两千行召回算子同时做 RPC、协议转换、过滤、特征处理和兜底，会导致任何变化都修改同一文件。拆成小算子后，每个算子的输入、输出和失败边界更清楚。

### 2.5 开闭原则是什么？

软件实体应对扩展开放、对修改关闭。例如新增算子主要通过实现 `Operator` 并注册到图中完成，而不是反复修改巨大 `if-else` 主流程。它不是绝对禁止修改，而是把稳定框架和变化业务分开。

### 2.6 依赖倒置是什么？

高层业务依赖抽象而不是具体实现。例如 `DynamicConfigStore` 依赖 `ConfigJournal`，生产使用文件实现，测试可用内存或故障实现；召回编排依赖 `RecallService`，可以切换本地实现和 gRPC 实现。

### 2.7 组合为什么通常优于继承？

组合能在运行时拼装能力，耦合更小。项目用装饰器给召回服务增加故障注入和韧性能力，用 `SecuredAdminHttpHandler` 包装原 Handler 增加鉴权，而没有建立复杂继承层级。

### 2.8 项目用了哪些设计模式？

| 模式 | 项目示例 | 作用 |
| --- | --- | --- |
| 策略模式 | `Operator`、不同召回服务 | 统一接口下替换算法 |
| 模板/框架 | `OperatorExecutor` | 固定执行、计时、异常处理流程 |
| 适配器 | 多个 `ProtoAdapter` | 外部 PB 转内部模型 |
| 工厂 | `RecallTransportFactory` | 根据配置选择 local/grpc |
| 装饰器 | `ResilientRecallService`、`SecuredAdminHttpHandler` | 为对象叠加能力 |
| 门面 | `RecommendationFacade` | 为 HTTP 层提供统一入口 |
| 注册表 | `MetricsRegistry`、`ResilienceRegistry` | 集中保存运行状态 |
| 快照 | `RuntimeConfigSnapshot` | 不可变版本配置 |

## 3. Java 基础与项目中的使用

### 3.1 接口和抽象类怎么选？

接口表达能力契约，支持多实现和组合；抽象类适合共享状态与公共实现。项目的 `Operator`、`RecallService`、`ConfigJournal` 用接口；`AbstractGrpcRecallService` 用抽象类复用 Channel、deadline 和调用框架。

### 3.2 为什么配置快照适合不可变对象？

不可变对象创建后状态不变，可以安全共享，减少线程间可见性和半更新问题，也便于审计和回滚。更新配置时创建 V2，而不是在 V1 对象上逐字段修改。

### 3.3 `enum` 相比字符串有什么优势？

枚举限制合法取值、支持编译期检查和 IDE 重构。字符串容易拼错并把错误推迟到运行时。项目用 `AttrName`、`DegradationLevel`、`FaultMode`、`PipelineVersion` 收敛硬编码。

### 3.4 `HashMap` 平均复杂度为什么是 O(1)？

通过 key 的 hash 定位桶，理想情况下只需常数次访问。发生碰撞后需要在桶内比较；Java 8 中碰撞严重时链表可树化，因此最坏情况不等于永远 O(1)。正确的 `equals/hashCode`、合理容量和分布良好的 hash 很重要。

### 3.5 为什么 `List<ItemAttr>` 查属性是 O(n)，`Map<AttrName, ?>` 是平均 O(1)？

List 不知道目标位置，需要从头比较；Map 根据 hash 直接定位桶。若一次请求对 N 个 Item 各查 M 个属性，列表方案可能接近 O(N×M×属性数)，Map 能明显降低热路径 CPU。

### 3.6 `AtomicReference` 能解决什么？

它能原子读取或替换一个对象引用，并提供必要的内存可见性。配置中心通过一次替换完整不可变快照，读线程不会看到“比例已更新、降级级别还没更新”的半状态。

项目落点：`DynamicConfigStore.current`。

### 3.7 `synchronized` 在配置仓库中有什么作用？

更新操作包含检查当前版本、落盘、替换引用和维护历史，属于一个复合临界区。仅靠 `AtomicReference` 不能自动保证整个复合流程互斥，所以 `update()` 和 `history()` 使用同步保护。

### 3.8 `volatile`、`synchronized`、原子类有什么区别？

- `volatile`：保证可见性和一定的有序性，不保证复合操作原子性；
- `synchronized`：提供互斥、可见性和可重入锁；
- 原子类：用 CAS 高效实现单变量原子更新，但复杂不变量仍需锁或不可变设计。

### 3.9 什么是 CAS 和 ABA 问题？

CAS 比较内存值是否仍是预期值，相同才替换。ABA 指值从 A 变 B 又变 A，CAS 看不出中间变化。可通过版本号解决；项目动态配置的 `expectedVersion` 本质上就是业务层版本检查。

### 3.10 checked exception 和 unchecked exception 怎么选？

checked exception 强制调用方处理，适合可恢复的外部条件；unchecked exception 适合编程错误或跨层统一处理。项目把参数错误映射为 400、版本冲突映射为 409、持久化失败映射为 503，重点不是继承体系本身，而是让调用方得到正确语义。

### 3.11 try-with-resources 为什么重要？

它保证 `AutoCloseable` 资源在正常和异常路径都关闭，避免文件描述符、连接和 Channel 泄漏。项目文件 Journal 使用 try-with-resources 管理 `FileChannel`。

### 3.12 Java 线程池的核心参数是什么？

核心线程数、最大线程数、存活时间、工作队列、线程工厂和拒绝策略。CPU 密集任务线程数通常接近 CPU 核数；I/O 密集任务可更多，但必须结合下游容量、连接数和超时测试，不能套固定公式。

### 3.13 为什么不能每个请求都 `new Thread`？

线程创建和栈内存成本高，无界创建会导致调度抖动和 OOM。线程池复用线程并提供队列与拒绝策略，但线程池本身也必须有界。

### 3.14 JVM 运行时内存区域有哪些？

- 堆：对象和数组的主要存储区域，线程共享；
- Java 虚拟机栈：每个线程独享，保存栈帧、局部变量和方法调用；
- 程序计数器：记录当前线程执行位置；
- 本地方法栈：服务 native 方法；
- 方法区的 HotSpot 实现 Metaspace：类元数据等，使用本地内存。

线程池过大不只消耗 CPU，每个线程栈也会占内存；缓存无界、日志队列无界和 nonce 集合不清理都可能推高内存。

### 3.15 堆溢出和栈溢出有什么区别？

大量对象长期可达可能导致 `OutOfMemoryError: Java heap space`；递归过深通常导致 `StackOverflowError`；线程过多还可能因无法创建 native thread 而 OOM。排查要结合错误类型、heap dump、线程 dump 和监控。

### 3.16 什么对象可以被 GC？什么是 GC Roots？

从 GC Roots 不可达的对象可以回收。常见 Roots 包括线程栈中的引用、静态字段、JNI 引用和正在运行的线程。项目 single-flight Map 如果完成后不删除 Future，即使业务不再使用，对象仍被 Map 引用，GC 也不能回收。

### 3.17 G1 的基本思路是什么？

G1 把堆划分为多个 Region，按收益优先回收，目标是兼顾大堆吞吐与可控停顿。Young GC 回收年轻代 Region，Mixed GC 还会回收部分老年代 Region。项目容器 JVM 使用 G1，但具体堆大小和暂停目标仍需根据监控压测调整。

### 3.18 什么是 Stop-The-World？

某些 GC 阶段需要暂停所有 Java 应用线程。STW 过长会直接表现为接口长尾。排查 P99 抖动时要把 GC 日志、CPU、线程池和下游 Trace 放在同一时间线分析。

### 3.19 JIT 和 warm-up 有什么关系？

JVM 会根据热点统计把字节码编译成本地机器码，并进行内联等优化。刚启动时类加载和编译尚未稳定，因此压测需要 warm-up；但生产冷启动和扩容阶段仍需单独评估。

### 3.20 类加载过程是什么？双亲委派有什么意义？

类经历加载、验证、准备、解析和初始化。双亲委派优先让父加载器尝试加载，避免核心类被应用随意替换，并保证同一类的基本唯一性。SPI 是一个典型例外场景，常通过线程上下文类加载器找到第三方实现。

### 3.21 `ThreadLocal` 有什么用途和风险？

它为每个线程保存独立变量，常用于请求上下文。但线程池会复用线程，如果请求结束不清理会串数据或内存泄漏；异步切换线程时值也不会自然传播。OpenTelemetry Context 应使用框架提供的传播机制，而不是假设 ThreadLocal 自动跨 Future/RPC。

## 4. JUnit、Mockito 与 Groovy

### 4.1 JUnit 是什么？

JUnit 是 Java 测试框架，负责组织测试生命周期、执行测试和断言结果。你的理解基本正确：测试自己构造输入，调用被测代码，再断言输出或异常是否符合预期。

```java
@Test
void shouldRejectInvalidPercent() {
    assertThrows(IllegalArgumentException.class,
        () -> new RuntimeConfigSnapshot(2, 101, 0, NONE, "me", Instant.now()));
}
```

### 4.2 Mockito 是什么？

Mockito 用来创建依赖的替身。它不是“再调用一个真实函数”，而是让测试提前规定依赖返回什么或抛什么，从而只验证当前单元。

```java
UserFeatureService dependency = mock(UserFeatureService.class);
when(dependency.get(123L)).thenReturn(new UserFeature(...));

PrepareOperator operator = new PrepareOperator(dependency, ...);
operator.execute(context);

verify(dependency).get(123L);
```

### 4.3 Mock、Stub、Spy 有什么区别？

- Stub 关注“调用时返回什么”；
- Mock 还关注“是否按预期交互”；
- Spy 包装真实对象，默认调用真实方法，只覆盖部分行为。

面试中不必死抠术语，说明你要隔离网络、数据库、时钟、随机数和故障即可。

### 4.4 单元测试和集成测试有什么区别？

单元测试范围小、快、依赖可替换；集成测试验证多个组件以及真实协议是否协作。`RecallOperatorTest` 属于单元测试，`GrpcRecallIntegrationTest` 启动真实 gRPC Server/Client，属于集成测试。

### 4.5 测试应该验证实现细节吗？

优先验证可观察行为，而不是私有方法或内部调用顺序。过度绑定实现会让正常重构导致大量测试失败。只有交互本身属于契约时才使用 `verify`。

### 4.6 如何测试超时、磁盘失败和时间窗口？

把外部不确定性抽象成依赖：固定 `Clock` 测时间；故障 `ConfigJournal` 测磁盘写失败；Mock 服务延迟或抛异常测超时和 fallback。这样测试稳定且不需要真的改系统时间或损坏磁盘。

### 4.7 Groovy 是什么？

Groovy 是运行在 JVM 上的动态语言，语法比 Java 简洁，能调用 Java 类，常用于脚本、构建 DSL 和动态规则。原 OnX 图中 Groovy 脚本便于快速写逻辑，但缺点是类型检查弱、重构困难、运行错误更晚暴露、监控和单测不统一。

### 4.8 为什么把 Groovy 脚本迁移成 Java 算子？

获得编译期类型检查、统一生命周期、依赖注入、指标、日志和单测，降低线上脚本错误风险。代价是修改需要编译发布，因此成熟系统可能保留受限 DSL，而不是允许任意脚本。

项目落点：`docs/groovy-script-example.groovy` 只用于学习对比，当前核心执行链不依赖 Groovy。

## 5. Context、数据结构与类型安全

### 5.1 `Map<String, Object>` 为什么危险？

- key 是散落的硬编码字符串，拼写错误只能运行时发现；
- value 是 Object，需要强转，容易 `ClassCastException`；
- 参数、特征和中间结果混放，生命周期和所有权不清；
- 大对象可能反复解析；
- 任意算子都能覆盖其他算子的值。

### 5.2 项目如何拆 Big Map？

`RecommendContext` 按用途保存请求、AB 参数、地址、特征、召回结果、混排结果和算子结果；属性名称使用枚举；不同来源的特征可用外层“来源”、内层“属性名”的 Map 表达。

### 5.3 强类型 Context 会不会变成另一个大对象？

会有这种风险。改进方向是按阶段拆分只读输入和输出类型，或使用带类型 key 的上下文。当前实现适合作为从 Big Map 到类型安全的中间阶段，不能宣称已经是最终完美模型。

### 5.4 为什么中间结果要有专门结构？

算子间通信需要明确生产者、消费者和生命周期。专门结构能减少 key 冲突，支持泛型检查，也方便记录耗时、状态和失败原因。

### 5.5 DTO、领域对象和 PB 为什么要分开？

DTO 面向接口传输，PB 面向跨进程协议，领域对象面向业务计算。直接让核心算法依赖外部 PB，会把协议演进影响扩散到全部算子。适配器隔离变化，代价是多一次转换和对象分配。

## 6. 算子框架与 DAG

### 6.1 什么是算子？

算子是一个边界清晰的处理单元，约定名称、配置和执行方法。它读取 Context 中的已知输入，产生明确输出，例如准备、召回、在线特征、过滤、混排和后处理。

项目落点：`Operator`、`OperatorConfig`、`OperatorExecutor` 和 `service/operator/impl`。

### 6.2 算子框架解决了什么？

- 统一参数配置；
- 统一计时、日志、指标和异常处理；
- 统一中间结果保存；
- 让业务逻辑从流程控制中分离；
- 为图调度、并行和测试提供标准接口。

### 6.3 什么是 DAG？为什么不是普通链表？

DAG 是有向无环图。链表只能表达 A→B→C；DAG 可以表达 B 和 C 都依赖 A、D 同时依赖 B/C。它适合描述存在分支和汇合的推荐流程，又因为无环所以能确定终止并进行拓扑调度。

### 6.4 什么是拓扑排序？

对 DAG 节点排序，使每条边的起点都在终点之前。Kahn 算法维护每个节点入度，把入度为 0 的节点放入队列，弹出后减少后继入度。最终处理节点数小于总数说明存在环。复杂度 O(V+E)。

### 6.5 如何发现 DAG 中的环？

可以使用 Kahn 算法处理数量判断，也可以 DFS 使用白/灰/黑三色标记；遇到指向灰色节点的边就是回边。

### 6.6 并行 DAG 如何调度？

依赖计数降为 0 的节点进入就绪队列，由有界线程池执行。节点完成后原子减少后继节点剩余依赖；全部前置完成的后继才能调度。汇合节点必须等待所有依赖，而不是等待任意一个。

项目落点：`DagGraph`、`DagNode`、`DagOperatorExecutor`、`ParallelDagOperatorExecutor`。

### 6.7 DAG 节点失败怎么办？

需要先定义语义：关键节点失败可以终止请求；非关键节点可降级并让图继续；下游节点可以跳过或读取 fallback。不能让框架偷偷吞异常。项目的韧性包装器把部分召回失败转为明确 fallback 结果。

### 6.8 图配置化有什么风险？

错误依赖、环、节点名称拼错、参数类型错误以及运行时才发现配置问题。生产系统需要发布前静态校验、版本管理、灰度、回滚和图可视化。

## 7. 并发、异步与关键路径

### 7.1 并发和并行有什么区别？

并发表示多个任务在同一时间段推进；并行表示同一时刻在多个执行单元上运行。单核可以并发但未必物理并行。

### 7.2 `CompletableFuture` 解决什么？

它表示未来完成的结果，支持组合、汇合、异常恢复和异步执行。三路召回可以分别提交，再通过 `allOf` 或逐个受控等待汇合。

### 7.3 `get()` 和 `join()` 有什么区别？

`get()` 抛 checked exception，支持带超时版本；`join()` 抛 unchecked `CompletionException`。生产链路通常需要明确 deadline，不能无界 `join()`。

### 7.4 为什么每个 RPC 超时之外还要 fan-out 总超时？

单调用超时限制某一路，fan-out 总预算限制整个阶段。重试、调度延迟或多个等待叠加可能让总耗时超过用户预算，因此需要分层 deadline。

### 7.5 并行三路召回的耗时怎么算？

理想情况下接近最慢一路加调度/合并开销，而不是三路之和。但如果线程池不足、连接池拥塞、CPU 饱和或下游互相争抢资源，收益会下降。

### 7.6 什么是竞态条件？

程序结果依赖不可控的线程执行顺序。例如两个配置更新都读取 V1 并各自写 V2。项目用同步临界区和 expectedVersion 避免丢失更新。

### 7.7 什么是死锁？四个必要条件是什么？

互斥、占有且等待、不可抢占、循环等待。避免方法包括固定加锁顺序、缩小锁范围、超时获取和减少嵌套锁。

### 7.8 什么是线程安全？

多个线程并发调用时，程序仍满足业务不变量。线程安全不只是“没有异常”，还包括不丢更新、不读半状态、顺序和统计符合定义。

### 7.9 Amdahl 定律对项目有什么启发？

整体加速受不可并行部分限制。如果只有 40% 链路可并行，即使该部分无限加速，整体最多约 1/(1-0.4)=1.67 倍。因此应先用 Trace 找关键路径，而不是盲目把所有算子异步化。

## 8. Protobuf、HTTP 与 gRPC

### 8.1 Protobuf 是什么？

Protobuf 是基于 schema 的二进制序列化协议，字段有编号和类型，体积通常小、解析快、支持跨语言代码生成。它不是 RPC；gRPC 可以用 Protobuf 作为消息格式。

### 8.2 为什么不同召回 PB 要转统一内部 PB？

不同下游的字段名、ID、分值和属性结构不同。如果转换散落在业务算子中，核心逻辑会依赖每个外部协议。适配器先收敛为 `InternalItemPb`，核心逻辑只处理统一语义，返回上游前再转换。

### 8.3 PB 字段号为什么不能随便改？

二进制数据通过字段号识别字段。删除字段后应 `reserved`，不能把原编号复用成新含义，否则旧数据可能被错误解释。新增可选字段一般向后兼容，旧端会忽略未知字段。

### 8.4 `repeated` 和 `map` 的差异是什么？

`repeated` 保留顺序并允许重复，查指定属性需要遍历；`map` 面向 key-value 快速查找且 key 唯一。协议选择应根据语义，而不只是性能。

### 8.5 gRPC 相比 REST 有什么特点？

gRPC 基于 HTTP/2 和强类型 IDL，支持多路复用、流式调用、deadline、metadata 和代码生成；REST/JSON 更易被浏览器和人工调试。内部高性能 RPC 常用 gRPC，对外开放接口常用 REST，两者不是绝对替代关系。

### 8.6 什么是 gRPC Channel 和 Stub？

Channel 管理到目标地址的连接和负载均衡状态，通常应长期复用；Stub 是基于 Channel 的类型安全客户端代理。每次请求新建 Channel 会造成握手、连接和资源开销。

### 8.7 deadline 和 timeout 有什么区别？

timeout 常表示允许持续多久，deadline 表示最晚完成时间点。跨服务传播绝对或剩余 deadline，可以避免每一跳都重新获得完整超时预算导致总耗时失控。

### 8.8 metadata 用来做什么？

传递不属于业务消息体的上下文，如 requestId、trace 上下文、认证信息。项目通过 metadata 传播 requestId 和 Trace Context。

### 8.9 gRPC Health Checking 和 Reflection 有什么用？

Health Checking 提供标准健康协议，供编排和探针判断服务；Reflection 让调试工具在没有本地 proto 时发现服务描述，生产是否开启要考虑暴露面。

### 8.10 为什么 fat JAR 需要合并 `META-INF/services`？

多个依赖可能通过 Java SPI 在 `META-INF/services` 注册实现。简单覆盖会丢失 provider，导致 gRPC 传输等功能运行时找不到。Shade 插件使用 `ServicesResourceTransformer` 合并这些文件。

项目落点：`pom.xml` 的 Maven Shade 配置。

### 8.11 TCP 三次握手解决什么？

客户端发送 SYN，服务端回复 SYN+ACK，客户端再回复 ACK，从而确认双方收发能力并协商初始序列号。远程 RPC 首次建连有握手成本，所以 gRPC Channel 应长期复用。

### 8.12 TCP 为什么可靠？

通过序列号、确认应答、校验、重传、滑动窗口、流量控制和拥塞控制提供有序字节流。可靠不等于调用一定成功：连接可能中断，客户端超时时服务端可能已经执行，因此仍要考虑幂等。

### 8.13 HTTP/1.1 和 HTTP/2 的主要区别是什么？

HTTP/1.1 通常依赖多个连接提高并发，同一连接存在队头阻塞；HTTP/2 使用二进制帧，在一个 TCP 连接中多路复用多个 Stream，并支持 Header 压缩。gRPC 依赖 HTTP/2 的多路复用和流能力。TCP 层丢包仍可能影响同连接多个 Stream。

### 8.14 长连接有什么优缺点？

优点是复用握手和连接资源、降低延迟；缺点是需要保活、空闲回收、断线重连、负载均衡和连接老化策略。长连接不代表永远不会断。

### 8.15 DNS 和服务发现是什么关系？

DNS 把服务名解析为地址。Compose 中服务名可作为容器 DNS 名；Kubernetes Service 也提供稳定 DNS。实例变化后客户端还要处理 DNS 缓存、连接重建和负载均衡。

### 8.16 TLS 提供什么？

TLS 提供传输加密、完整性和服务端身份认证，mTLS 还认证客户端。HMAC 不能替代 TLS：HMAC 防伪造，但未加密的请求内容仍可能被窃听。

## 9. 分布式系统与稳定性治理

### 9.1 分布式系统的典型问题有哪些？

网络延迟和分区、消息丢失/重复/乱序、时钟不一致、部分失败、重试放大、服务发现、数据一致性和可观测性。不能因为本机函数调用成功，就假设远程调用可靠。

### 9.2 什么是超时？为什么必须有？

超时给等待设置上界，释放线程、连接和请求预算。没有超时的慢调用会占满资源并造成级联故障。超时值应基于下游延迟分布和上游总预算，而不是拍脑袋。

### 9.3 什么请求可以重试？

通常是幂等操作和明确的瞬时错误。参数错误、权限失败和业务拒绝不应重试。重试需要次数上限、退避、抖动和总 deadline，避免重试风暴。

### 9.4 什么是幂等？

同一操作执行一次和多次的最终效果相同。GET 通常应幂等；配置更新用 expectedVersion 防止同一旧版本重复覆盖，但完整生产发布还可以增加幂等请求 ID。

### 9.5 熔断器三种状态是什么？

- CLOSED：正常放行并统计失败；
- OPEN：达到失败阈值后快速失败；
- HALF_OPEN：等待窗口后放少量探测请求，成功则关闭，失败则重新打开。

项目落点：`CircuitBreaker`、`CircuitState`。

### 9.6 熔断和限流有什么区别？

限流保护自己或整体容量，控制进入量；熔断根据某个依赖的失败情况停止继续调用它。两者可以同时使用。

### 9.7 什么是 Bulkhead（舱壁隔离）？

像船舱隔板一样为不同依赖分配独立并发容量。live 卡死时不能占满 goods/ad 的线程。常用独立线程池、信号量或连接池实现。

### 9.8 什么是 fallback？

主路径失败后的可接受替代结果，例如 live 失败时返回 goods+ad 的 17 条候选。fallback 保可用性，但可能降低推荐质量，因此必须统计 fallback 率，不能只看 HTTP 200。

### 9.9 什么是降级？

系统主动关闭非核心能力或降低计算复杂度以保核心服务，例如减少返回数量、跳过昂贵步骤。降级可以手动、配置驱动或根据指标自动触发。

### 9.10 熔断、降级、兜底的关系是什么？

熔断决定是否继续调用故障依赖；降级决定系统采用较轻策略；兜底提供替代结果。它们分别作用于调用决策、能力等级和输出结果。

### 9.11 什么是故障注入？

主动制造超时、错误或宕机，以验证保护机制是否真的工作。项目既有进程内 `FaultInjectionManager`，也会真实停止 Docker 容器，后者能覆盖网络连接和进程生命周期问题。

### 9.12 为什么“服务还能返回”不等于容灾通过？

还要验证返回数量和质量、延迟、错误类型、熔断状态、告警以及恢复后能否回到正常结果。项目最终验收验证 live 故障为 25→17→25。

## 10. Redis 与缓存

### 10.1 Cache Aside 模式是什么？

读流程：先查缓存，命中直接返回；未命中查数据源，再写缓存。写流程通常先更新数据源，再删除缓存。项目缓存用户特征，miss 时回源 `UserFeatureService`。

### 10.2 缓存穿透、击穿、雪崩分别是什么？

- 穿透：查询不存在数据，缓存永远 miss，持续打到数据源；
- 击穿：热点 key 过期，大量并发同时回源；
- 雪崩：大量 key 同时过期或 Redis 整体故障，大量请求同时回源。

### 10.3 项目如何应对？

- 穿透：空值哨兵并设置短 TTL；
- 击穿：`CompletableFuture` single-flight 合并同 key 并发回源；
- 雪崩：TTL 加随机抖动；
- Redis 故障：fail-open，直接回源并记录 error 指标。

项目落点：`CachedUserFeatureService`、`RedisFeatureCacheClient`。

### 10.4 为什么 TTL 要加随机抖动？

若一批 key 同时写入并使用相同 TTL，会集中失效。随机抖动把过期时间摊开，降低瞬时回源峰值。

### 10.5 什么是 single-flight？

同一个 key 在同一时刻只允许一个请求执行回源，其他请求等待同一个 Future。它减少热点 miss 放大，但必须在完成或异常时移除 Future，避免永久卡住或内存泄漏。

### 10.6 缓存为什么采用 fail-open？

缓存只是加速层，Redis 失败时仍可访问真实特征服务，所以选择绕过缓存保业务可用。代价是数据源压力上升，因此需要连接超时、告警、限流和容量预案。

### 10.7 缓存一致性怎么保证？

强一致代价高，Cache Aside 通常是最终一致。常见策略是更新数据库后删除缓存，处理并发时可结合版本号、延迟双删、CDC 或消息队列。本项目数据是 Demo 特征，重点展示读取路径与故障保护，没有实现真实写一致性。

### 10.8 Redis 为什么快？

内存访问快，核心命令执行模型减少锁竞争，使用高效数据结构和 I/O 多路复用。不能简单回答“因为单线程”；现代 Redis 的网络读写等部分也可多线程，且慢命令、大 key、持久化和网络仍会造成延迟。

## 11. 动态配置、版本与持久化

### 11.1 为什么配置更新使用乐观锁？

发布方携带 `expectedVersion`。只有当前版本相同才产生下一版本，否则返回 409。这样两个发布者同时基于 V3 修改时，后提交者不会静默覆盖前者。

### 11.2 乐观锁和悲观锁区别是什么？

悲观锁先独占资源再修改，冲突时等待；乐观锁允许并发读取，提交时校验版本，冲突后重试或人工合并。配置发布写入少、读取多，适合乐观锁。

### 11.3 配置为什么采用拉取而不是推送？

拉取实现简单，实例可在重启后自主恢复，配置中心无需维护大量长连接；代价是存在轮询延迟和额外请求。生产系统常用推送通知加定期拉取校准。

### 11.4 什么是 Last Known Good？

最后一次成功校验并应用的配置。配置中心不可用或返回非法数据时，网关继续使用 LKG，而不是清空配置或阻断推荐。`/runtime-config` 会显示 HEALTHY/STALE、版本和错误数。

### 11.5 为什么配置必须是完整快照而不是逐字段更新？

多个字段可能共同构成业务不变量。完整快照经过统一校验后原子替换，避免读到半更新状态，也便于审计、回滚和 Diff。

### 11.6 V21 为什么先落盘再更新内存？

顺序是校验版本→构造快照→追加日志并 `force(true)`→更新内存→返回成功。若先更新内存，磁盘失败后调用方可能看到一个重启即消失的假成功版本。

### 11.7 `fsync`/`force(true)` 解决什么？

普通 write 可能只进入操作系统缓存。force 请求把数据同步到持久化设备，提高宕机后可恢复概率。它不能代替磁盘冗余、备份和一致性协议。

### 11.8 什么是 WAL/追加日志？

先把变更按顺序追加到日志，再更新内存状态；恢复时重放日志。顺序写简单且保留历史。项目用 JSON Lines 实现教学版 Journal，每个配置版本一行。

### 11.9 为什么只修复损坏的最后一行？

宕机可能打断当前追加，因此只有末行截断符合正常崩溃模型。中间行损坏或版本断档意味着更严重的数据问题，自动跳过会掩盖事故，所以拒绝启动。

### 11.10 文件 Journal 能否支持配置中心多副本？

不能直接支持。多个进程并发写同一文件缺乏选主和一致性协议。生产多副本应使用数据库、etcd、ZooKeeper、Raft 系统或成熟配置中心，并设计备份恢复。

## 12. 灰度发布、AB 实验与影子流量

### 12.1 灰度发布是什么？

先让少量稳定流量进入新版本，观察正确性、延迟和业务指标，再逐步放量。异常时快速回滚，降低一次全量发布的爆炸半径。

### 12.2 为什么按用户稳定分桶？

同一用户应稳定进入同一版本，否则体验和实验数据会抖动。常见做法是对 userId 与实验 salt 做 hash，再对 100 或 10000 取模。

### 12.3 AB 实验和技术灰度有什么区别？

技术灰度主要验证稳定性和兼容性；AB 实验比较策略对业务指标的因果影响，需要随机分组、互斥层、样本量和统计显著性。两者都可用分桶，但目标和分析方法不同。

### 12.4 什么是 shadow traffic？

真实请求主链路正常返回，同时异步复制给新链路执行，新链路结果不影响用户，只做 Diff 和性能观察。它适合上线前验证，但会增加下游负载，并要处理用户隐私和副作用。

### 12.5 影子链路为什么必须无副作用？

如果影子请求会扣库存、写订单或重复计费，双跑会产生真实错误。需要只读接口、dry-run 标记或隔离存储。

### 12.6 推荐结果 Diff 看哪些指标？

Item 数量、顺序、分值差、Top-K 重合率、来源占比、耗时和异常。完全相等通常过于严格，应按业务允许范围定义门槛。

项目落点：`RecommendationDiffEngine`、`ComparisonRegistry`、`MigrationRecommendationFacade`、`RolloutManager`。

### 12.7 为什么重构完成两个月，推全却花更久？

代码完成不等于行为等价。推荐系统对排序、过滤、随机性和数据时效敏感，小差异可能影响业务指标。需要影子打平、灰度观察、定位差异和逐步放量，发布风险控制往往比编码更耗时。

## 13. 可观测性：日志、指标、Trace 和告警

### 13.1 可观测性三支柱是什么？

- Logs：离散事件及上下文；
- Metrics：可聚合的数值时间序列；
- Traces：一次请求跨服务的因果链路。

三者互补：指标发现异常，Trace 定位慢在哪个服务，日志查看具体错误细节。

### 13.2 结构化日志比字符串日志好在哪里？

字段固定、机器可解析，可以按 requestId、operator、source、costMs 和 errorType 查询聚合。生产日志要避免密码、Token、完整地址等敏感信息。

项目落点：`StructuredLogger`、`LogLevel`。

### 13.3 lazy logging 有什么意义？

日志级别未开启时，不提前拼接昂贵字符串或序列化大对象。延迟发送可把 I/O 从主链路移开，但要有有界队列、丢弃策略和退出刷盘，否则日志系统会反过来拖垮业务。

### 13.4 Counter、Gauge、Histogram 分别是什么？

- Counter：只增累计值，如请求数、错误数；
- Gauge：可增可减瞬时值，如队列长度、并发数；
- Histogram：把观察值分桶，适合延迟和响应大小，可估算分位数。

### 13.5 为什么不能给 Prometheus label 放 userId/requestId？

这会产生海量唯一时间序列，造成高基数，消耗内存和存储并拖慢查询。高基数标识应进入日志或 Trace；指标 label 应使用有限集合，如 scene、source、status。

### 13.6 P50、P95、P99 是什么？

P95 表示 95% 请求耗时不超过该值。平均值会掩盖长尾，推荐链路通常更关注 P95/P99。分位数必须说明统计窗口、样本量和客户端/服务端口径。

### 13.7 TraceId、SpanId 是什么？

TraceId 标识整条请求链，SpanId 标识其中一次操作或 RPC。父子 Span 表达因果关系。跨线程和 gRPC 时必须显式传播 Context，否则 Trace 会断裂。

项目落点：`TracingHttpHandler`、`Telemetry`、`GrpcTelemetry`、`TraceCollectorApplication`。

### 13.8 告警应该怎么设计？

告警应可行动，围绕用户影响和 SLO：高错误率、高 P95、fallback 率、配置长时间 STALE、缓存错误和下游不可用。需要持续窗口抑制瞬时毛刺，并有恢复通知和负责人。

### 13.9 健康检查和业务监控一样吗？

不一样。健康检查回答实例能否接流量；业务监控判断效果和质量是否异常。服务返回 200 但 fallback 率 50%，健康探针可能通过，业务 SLO 却失败。

## 14. Docker 与容器化

### 14.1 镜像和容器的区别是什么？

镜像是不可变文件系统和启动配置；容器是镜像的运行实例，有进程、网络和可写层。同一镜像可启动多个容器。

### 14.2 Dockerfile 的核心指令有哪些？

`FROM` 基础镜像、`WORKDIR` 工作目录、`COPY` 复制产物、`ENV` 环境变量、`USER` 运行用户、`ENTRYPOINT/CMD` 启动命令。

### 14.3 为什么容器应使用非 root 用户？

降低容器逃逸或应用漏洞后的权限。项目主应用使用 UID 65532，并设置只读根文件系统；配置 Volume 由一次性 init 容器调整目录所有权。

### 14.4 为什么要只读根文件系统？

减少恶意或错误写入镜像层的机会，让持久化位置显式化。临时文件放 tmpfs，配置日志放命名卷。

### 14.5 Volume 和 bind mount 区别是什么？

Volume 由 Docker 管理，适合持久数据；bind mount 映射宿主机明确路径，适合本地配置和开发。项目配置 Journal 和监控数据用命名卷，Prometheus/Grafana 配置用只读 bind mount。

### 14.6 Docker Compose 的作用是什么？

声明多容器服务、网络、依赖、健康检查、端口、资源和 Volume，让本地分布式拓扑可一键复现。它不是完整生产编排平台。

### 14.7 `depends_on` 能保证业务完全可用吗？

仅声明启动依赖时不能；结合 `condition: service_healthy` 可等待健康检查，但仍不能替代应用自身重试和容错，因为运行过程中依赖还会再次故障。

## 15. Kubernetes

### 15.1 Pod、Deployment、Service 分别是什么？

- Pod：Kubernetes 最小调度单元；
- Deployment：管理无状态 Pod 副本、滚动更新和回滚；
- Service：为一组 Pod 提供稳定虚拟地址和服务发现。

### 15.2 liveness、readiness、startup probe 区别是什么？

- startup：应用是否完成启动，成功前可保护慢启动应用；
- readiness：是否应接收流量，失败会从 Service Endpoint 摘除；
- liveness：是否需要重启进程。

错误地把下游短暂不可用作为 liveness 失败，可能造成重启风暴。

### 15.3 requests 和 limits 是什么？

requests 用于调度和资源保障，limits 是容器上限。CPU 超 limit 会节流，内存超 limit 可能 OOMKilled。必须根据压测和监控设置，而不是复制模板。

### 15.4 HPA 根据什么扩缩容？

可根据 CPU、内存或自定义指标调节副本数。推荐网关更适合结合 CPU、并发、队列或请求率；仅看 CPU 可能无法发现 I/O 等待造成的拥塞。

### 15.5 PDB 有什么作用？

PodDisruptionBudget 限制自愿驱逐时同时不可用的 Pod 数量，例如节点维护。它不能防止节点宕机、应用崩溃或所有非自愿故障。

### 15.6 滚动更新如何做到零不可用？

足够副本、正确 readiness、`maxUnavailable: 0`、合理 `maxSurge`、优雅终止和连接排空。所谓零不可用仍需真实流量验证。

### 15.7 ConfigMap 和 Secret 区别是什么？

ConfigMap 保存非敏感配置；Secret 保存敏感数据，但默认只是编码存储，仍需 RBAC、静态加密和外部密钥管理。V22 HMAC 密钥不应写入 Git。

## 16. 性能、压测、容量与 SLO

### 16.1 QPS、并发和响应时间有什么关系？

稳定系统中可用 Little's Law 近似：并发数 ≈ 吞吐量 × 平均响应时间。例如平均耗时 0.2 秒、QPS 10，并发约 2。排队和闭环压测会影响实际结果。

### 16.2 开环和闭环压测有什么区别？

闭环用户发完一个请求才发下一个，服务变慢时发压自动下降；开环按目标速率持续发请求，更容易暴露排队和过载。项目自带的是闭环并发压测器，应如实说明。

### 16.3 为什么需要 warm-up？

JIT 编译、类加载、连接建立和缓存预热会使初始请求偏慢。先预热再采样能更接近稳态，但冷启动性能仍应另测。

### 16.4 吞吐提高为什么不一定代表容量提高？

容量必须同时满足延迟、成功率和业务质量 SLO。项目并发 4 的 QPS 更高，但 fallback 率接近 50%，因此不能作为可用容量。

### 16.5 SLI、SLO、SLA 区别是什么？

- SLI：实际测量指标，如成功率、P95、fallback 率；
- SLO：内部目标，如成功率 ≥99.9%；
- SLA：对外承诺，违约可能有赔偿。

### 16.6 什么是错误预算？

允许失败量等于 1-SLO。例如 99.9% 可用性允许 0.1% 失败。错误预算消耗过快时应暂停高风险发布，优先稳定性建设。

### 16.7 为什么客户端和服务端延迟都要看？

服务端计时可能不含 DNS、连接、网络和客户端排队。用户体验应看客户端端到端延迟，服务定位再拆分服务端 Span。

### 16.8 如何找系统瓶颈？

逐级增加并发，观察 QPS 是否继续线性增长，同时查看 P95/P99、错误、fallback、CPU、内存、GC、线程池、连接池和下游延迟。瓶颈不是简单等于 CPU 最高的组件。

项目落点：`HttpLoadGenerator`、`LoadResultCalculator`、`CapacityBenchmarkApplication`。

## 17. 管理面安全

### 17.1 Authentication 和 Authorization 区别是什么？

认证回答“你是谁”，授权回答“你能做什么”。合法 OPS 身份调用发布接口时认证成功，但授权失败，返回 403。

### 17.2 HMAC 是什么？

HMAC 使用共享密钥和哈希构造消息认证码，验证消息来源和完整性。项目用 HMAC-SHA256 签名方法、路径、原始查询串、时间戳和 nonce，密钥本身不在网络上传输。

### 17.3 HMAC 和普通 SHA-256 有什么区别？

普通 SHA-256 没有密钥，攻击者修改消息后可重新计算 hash；HMAC 需要共享密钥，攻击者无法生成合法认证码。

### 17.4 为什么签名要覆盖 method、path 和 query？

否则攻击者可能把 GET 改 POST、换接口或修改灰度比例而复用签名。Canonical Request 必须定义编码和顺序，客户端与服务端完全一致。

### 17.5 为什么需要时间戳和 nonce？

签名只能证明请求合法，不能阻止复制。时间戳限制可接受窗口，nonce 保证窗口内唯一，从而防止重放。

### 17.6 为什么签名比较使用常量时间？

普通比较可能在第一个不同字节就返回，耗时差异理论上可泄露匹配前缀。`MessageDigest.isEqual` 降低时序攻击风险。

### 17.7 401 和 403 的区别是什么？

401 表示缺少或无法验证身份；403 表示身份已确认，但没有权限。400 是请求格式错误，409 是资源版本冲突，503 是服务暂不可用。

### 17.8 为什么状态变更必须使用 POST？

GET 应安全、只读、可缓存。浏览器预取、爬虫或监控访问 GET 不应改变线上状态。项目对带管理参数的 GET 返回 405。

### 17.9 当前 HMAC 方案的边界是什么？

多副本 nonce 未共享、密钥轮换未完善、缺少人员身份和审批、必须配合 HTTPS。生产可使用 API Gateway、mTLS、OIDC/JWT、IAM、Vault 和集中审计。

项目落点：`security` 包、`SecuredAdminHttpHandler`、`AdminRequestAuthenticator`。

## 18. Maven、构建、CI/CD 与 Git

### 18.1 Maven 生命周期常见阶段是什么？

`compile` 编译主代码，`test` 运行单测，`package` 生成包，`verify` 执行验证，`install` 安装到本地仓库，`deploy` 发布远端仓库。项目发布门禁使用 `mvn clean verify`。

### 18.2 dependency scope 有哪些？

`compile` 默认参与编译运行；`test` 只供测试；`provided` 编译需要但运行环境提供；`runtime` 编译不直接需要、运行需要。JUnit/Mockito 是 test scope。

### 18.3 fat JAR 是什么？

把应用和运行依赖合并为一个可执行 JAR，部署简单；代价是包更大、资源冲突和许可证文件重复。项目用 Maven Shade 插件生成。

### 18.4 CI 和 CD 分别是什么？

CI 持续集成：每次改动自动编译、测试和质量检查；CD 可指持续交付或持续部署，前者确保随时可发布，后者自动上线。当前 GitHub Actions 属于 CI/发布门禁，没有自动部署生产。

### 18.5 为什么依赖 Action 要固定完整 SHA？

Tag 可能被移动，完整提交 SHA 能锁定执行代码，降低供应链漂移风险。项目的 `checkout` 和 `setup-java` 已固定 SHA，并用注释标记主版本。

### 18.6 一个可靠发布门禁应包含什么？

编译、单测、集成测试、静态检查、依赖/镜像安全检查、镜像构建、配置校验、必要的性能回归和制品签名。当前项目覆盖测试和镜像构建，安全扫描与制品发布仍可继续增强。

### 18.7 Git 的 merge 和 rebase 区别是什么？

merge 创建合并提交并保留分支拓扑；rebase 把提交重放到新基线，历史更线性但会改写 commit SHA。共享分支不要随意 rebase 后强推。

## 19. 高频综合追问

### 19.1 为什么不用 Spring Boot？

项目刻意使用 JDK `HttpServer` 和手工 Wiring，让学习者看到 HTTP、依赖组装、线程池和生命周期的本质，避免框架自动配置遮住核心机制。真实生产项目可使用 Spring Boot 提升生态集成和开发效率。

### 19.2 为什么配置中心不用 MySQL/etcd？

教学版用 append-only 文件把持久化顺序、崩溃恢复和版本校验讲清楚。它适合单写者和低写入量，不支持生产多副本一致性；生产应选成熟存储。

### 19.3 为什么 Redis 故障不直接让健康检查失败？

因为缓存是可绕过依赖，主链路能回源。若 readiness 因 Redis 故障摘除全部网关，反而扩大故障。健康模型应区分关键依赖和可降级依赖。

### 19.4 为什么不无限重试？

无限重试占用线程和连接，增加下游压力并突破总 deadline。重试只能提高瞬时故障成功率，不能修复持续故障。

### 19.5 为什么有了熔断还要超时？

熔断需要失败信号来判断状态，超时给单次等待设上限；熔断打开后才会快速失败。两者解决不同阶段的问题。

### 19.6 为什么有了单测还要 Docker 故障实验？

单测验证类级逻辑，无法完全覆盖端口、DNS、容器退出、连接重建、Volume 权限和真实序列化。项目 V21 正是通过真实容器发现非 root 用户无权写新 Volume 的问题。

### 19.7 这个项目最大的亮点是什么？

不是技术数量，而是从复杂度问题出发，形成“抽象—并行—稳定性—可观测—发布—验收”的完整演进，并为每个关键结论提供单测、集成测试或真实故障证据。

### 19.8 这个项目还有哪些不足？

- 配置 Journal 仅支持单写实例；
- HMAC nonce 状态未跨副本共享；
- Demo 下游和混排不是真实算法服务；
- 压测是单机闭环，不能代表真实生产容量；
- 缺少真实数据库、消息队列、服务网格和多机房；
- CI 尚可增加依赖扫描、SBOM、镜像扫描与制品签名；
- Kubernetes 演示以本地集群为主，没有真实云上流量验证。

能够主动说明边界，会比声称“已达到大厂生产水平”更可信。

## 20. 三分钟项目介绍模板

> 我做的是一个电商推荐接入层工程化项目。接入层向上承接商城等流量，向下编排商品、直播、广告召回，完整链路包括参数和特征准备、多路召回、在线特征、过滤、混排和后处理。  
>  
> 项目最初模拟的是高复杂度旧代码：请求数据放在 `Map<String,Object>` 中，大算子和动态脚本导致耦合、类型不安全、定位和测试困难。我先把上下文拆成强类型结构，用枚举收敛属性 key，并把不同下游 PB 通过适配器转成统一 Item；然后抽象 Operator 执行框架，把流程建模为 DAG，通过依赖关系调度并行节点。  
>  
> 在性能方面，三路召回并行，并探索在线特征和混排并行，把串行耗时从相加变为取最大值。稳定性方面增加分层 deadline、有限重试、熔断、Bulkhead、fallback 和分级降级；live 服务宕机时结果从 25 条降为 goods+ad 的 17 条，恢复后回到 25 条。  
>  
> 为了安全发布，我实现稳定用户分桶、影子双跑和结果 Diff，并用独立配置中心动态控制灰度、shadow 和降级。配置使用 expectedVersion 乐观锁、网关保留 Last Known Good；后来增加 append-only Journal，先落盘再发布内存，真实重启容器后仍能恢复版本和审计记录。  
>  
> 工程侧还使用 Protobuf/gRPC 拆成多进程，通过 OpenTelemetry、Prometheus、Grafana 打通日志、指标、Trace 和告警，用 Docker Compose 与 Kubernetes 部署，并引入 Redis Cache Aside、TTL 抖动、空值缓存、single-flight 和 fail-open。最后通过容量 SLO、65 项测试、GitHub Actions 和 HMAC 管理面鉴权完成发布门禁。  
>  
> 我认为这个项目最有价值的部分不是堆技术，而是每次演进都有明确问题、方案、权衡和验证，同时我也清楚单机 Journal、单机 nonce 和本机压测与真实大规模生产系统之间的边界。
